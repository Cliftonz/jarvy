# Jarvy Implementation Progress

> **Note**: Jarvy is a **non-profit, open-source project**. It exists to help teams provision and standardize developer workstations without relying on VMs, virtualization, containers, or any of the commercial/proprietary solutions listed in `docs/competitors/`. The goal is simple: give developers a fast, native way to get their machines set up consistently across an organization.

> **Scope Note**: Jarvy currently supports **online installations only**. Offline/air-gapped features (bundle mode, cache warming, portable mode) are not covered at this time. PRD-025 is scoped to local mirror support for organizations with internal package repositories—users still need network connectivity to those mirrors.
>
> **Not Implemented**: The following features are explicitly out of scope:
> - **Programmatic/API Access** - No REST/GraphQL API, no embeddable library crate, no webhook integrations
> - **Cost & Resource Visibility** - No bandwidth tracking, disk space reporting, or download analytics
> - **Plugin/Extension Ecosystem** - No third-party tool definitions, plugin registry, or custom package manager extensions
> - **Parallel/Group Downloads** - No parallel binary downloads or download batching. Package managers (brew, apt, etc.) handle their own download optimization, and custom installers (rustup, nvm) already use efficient strategies. Complexity not justified for marginal benefit.

## PRD-004: Semantic Version Checking

**Files**: `src/tools/version.rs` (new), `src/tools/common.rs`
**Dependencies**: semver, regex, once_cell
**Tests**: 24 unit tests

**Key Features**:
- Semver operators: `>=`, `<`, `^`, `~`, `=` and ranges
- Smart version extraction from tool output (git, node, python, docker, rustc, go)
- Component-boundary matching prevents false positives ("3.1" won't match "3.10.0")

**Key Learnings**:
- `VersionReq::parse("3.10.5")` returns `^3.10.5`, not exact match - must use `=3.10.5`
- Regex pattern: `v?(\d+)\.(\d+)(?:\.(\d+))?(?:-([a-zA-Z0-9.-]+))?`

## PRD-001: Optimized Tool Installation (Phase 1 - Batch Operations)

**Files**: `src/tools/common.rs`, `src/tools/spec.rs`, `src/main.rs`
**Dependencies**: None (Phase 1 uses no new dependencies)
**Tests**: 10 new tests (batch_install_tests, tool grouping tests)

**Key Features**:
- `PackageManager` enum with 10 variants (Apt, Dnf, Yum, Zypper, Pacman, Apk, Brew, BrewCask, Winget, Choco)
- `batch_install()` method for batching package manager operations (`brew install a b c` instead of individual installs)
- `BatchInstallResult` struct for partial failure tracking (succeeded + failed lists)
- Tool grouping via `group_tools_for_installation()` to categorize by PM/custom installer/unknown
- Refactored setup flow: group tools → batch install by PM → install custom tools individually → run hooks

**Batch Install Commands by PM**:
| Package Manager | Batch Command |
|-----------------|---------------|
| Homebrew | `brew install a b c` |
| Homebrew Cask | `brew install --cask a b c` |
| apt | `apt install -y a b c` |
| dnf | `dnf install -y a b c` |
| yum | `yum install -y a b c` |
| zypper | `zypper install -y a b c` |
| pacman | `pacman -S --noconfirm a b c` |
| apk | `apk add a b c` |
| winget | `winget install -e --id a b c` (sequential internally) |
| Chocolatey | `choco install a b c -y` |

**Key Code Patterns**:
```rust
// Tool grouping
let groups = tools::spec::group_tools_for_installation(
    tools.iter().map(|(_, t)| (t.name.as_str(), t.version.as_str()))
);

// Batch install per PM
for (pm, packages) in &groups.by_package_manager {
    match tools::common::PkgOps::batch_install(*pm, &package_names, None) {
        Ok(result) => { /* handle succeeded/failed */ }
        Err(e) => { /* batch failed entirely */ }
    }
}

// Custom tools individually
for (name, version) in &groups.custom_install {
    tools::add(name, version)?;
}
```

**Key Learnings**:
- `PackageManager` enum needs `PartialEq, Eq, Hash` derives for HashMap key usage
- Batch failures must track both successful and failed packages for proper reporting
- `MANUAL_TOOLS` constant defines tools with custom installers (nvm, rust, brew) separately from `define_tool!` macro tools
- Tool grouping must distinguish between unknown tools (not in registry) and tools with custom installers
- Dry-run mode must show batch operations grouped by PM for UX clarity

**Performance Impact**:
- 10 brew tools: ~5 min → ~1 min (single dependency resolution, one lock acquisition)
- Eliminates redundant network requests for package index updates

**Follow-up PRD Created**: PRD-002 (Tool Post-Install Hooks) - identifies tools that could benefit from automatic shell integration/completion hooks

## PRD-001: Phase 2 - Parallel Version Checking

**Files**: `src/tools/spec.rs` (added ~140 lines)
**Dependencies**: rayon (1.10)
**Tests**: Existing tests pass (552 total)

**Key Features**:
- `ToolVersionStatus` struct: holds tool name, version, satisfied status, known status
- `VersionCheckSummary` struct: categorizes results into satisfied/needs_install/unknown with timing
- `check_tools_parallel()`: Uses `rayon::par_iter()` for concurrent version checking
- `check_tools_sequential()`: Fallback for debugging with `--sequential` flag

**Key Code Pattern**:
```rust
use rayon::prelude::*;

pub fn check_tools_parallel<'a, I>(tools: I) -> VersionCheckSummary
where
    I: Iterator<Item = (&'a str, &'a str)>,
{
    let tool_list: Vec<(&str, &str)> = tools.collect();
    let results: Vec<ToolVersionStatus> = tool_list
        .par_iter()
        .map(|(name, version)| check_tool_version(name, version))
        .collect();
    // Categorize results...
}
```

**Key Learnings**:
- HashMap iterators return `(&K, &V)` tuples - use `.map(|(k, v)| (k.clone(), v.clone()))` not `.cloned()`
- `define_tool!` macro requires ALL fields in linux section (apt, dnf, pacman, apk) - missing field causes "unexpected end of macro invocation"
- rayon's `par_iter()` requires collecting iterator into Vec first

**Performance Impact**:
- Version checks (10 tools): ~10 seconds → ~2 seconds (5x improvement)

**Shell Completion Pattern**:
```rust
default_hook: {
    description: "Install tool shell completions for bash and zsh",
    script: r#"
if [ -f "$HOME/.bashrc" ]; then
    mkdir -p "$HOME/.local/share/bash-completion/completions"
    tool completion bash > "$HOME/.local/share/bash-completion/completions/tool" 2>/dev/null || true
fi
if [ -f "$HOME/.zshrc" ]; then
    mkdir -p "$HOME/.zsh/completions"
    tool completion zsh > "$HOME/.zsh/completions/_tool" 2>/dev/null || true
    if ! grep -q 'fpath.*\.zsh/completions' "$HOME/.zshrc"; then
        echo 'fpath=($HOME/.zsh/completions $fpath)' >> "$HOME/.zshrc"
    fi
fi
"#
}
```

**Key Learnings**:
- Most CLI tools support `tool completion bash/zsh` for generating completions
- Some tools use different syntax: `glab completion -s bash`, `just --completions bash`, `rg --generate=complete-bash`
- zsh completions go to `~/.zsh/completions/_tool` (underscore prefix)
- bash completions go to `~/.local/share/bash-completion/completions/tool`
- Always add fpath modification check for zsh (`if ! grep -q 'fpath'...`)

- **Build provenance**: GitHub Actions attestation
- **Verification script**: `scripts/security/verify-signatures.sh`

### 7. OpenSSF Scorecard
- Weekly automated scoring
- Target: 8.0+ out of 10
- Checks: Branch protection, CI tests, code review, SAST, signed releases, etc.

```rust
// MCP Server main loop
pub fn run(mut self) -> McpResult<()> {
    let mut transport = StdioTransport::new();
    loop {
        match transport.read_request() {
            Ok(Some(request)) => {
                let response = self.handle_request(&request);
                transport.write_response(&response)?;
            }
            Ok(None) => break, // EOF
            Err(e) => {
                transport.write_response(&JsonRpcResponse::error(None, e))?;
            }
        }
    }
    Ok(())
}
```

**Key Learnings**:

1. **LinuxInstall has no `uniform` field**: The `uniform()` is a constructor method, not a field. Access individual package manager fields (apt, dnf, pacman, etc.) directly.

2. **Option<&'static str> chaining**: Use `.or()` directly without `.as_ref()` for chaining `Option<&'static str>` values:
   ```rust
   // Correct
   l.apt.or(l.dnf).or(l.pacman).unwrap_or("unknown")

   // Wrong (type mismatch)
   l.apt.as_ref().or(l.dnf.as_ref()).unwrap_or(&"unknown".to_string())
   ```

3. **Sliding window rate limiting**: Use `VecDeque<Instant>` for efficient O(1) timestamp management:
   ```rust
   fn check_limit(&mut self, limit: usize) -> Result<(), McpError> {
       let now = Instant::now();
       let window = Duration::from_secs(60);

       // Remove old timestamps
       while self.timestamps.front().is_some_and(|t| now.duration_since(*t) > window) {
           self.timestamps.pop_front();
       }

       if self.timestamps.len() >= limit {
           return Err(McpError::rate_limited());
       }

       self.timestamps.push_back(now);
       Ok(())
   }
   ```

## PRD-024: Team & Enterprise Collaboration (Specification Update)

**Date**: 2026-01-16
**Status**: Specification expanded with recursive extending details

**Added Section: Recursive Extending Resolution**

The PRD now comprehensively defines how config inheritance works when configs extend other configs that also use `extends`:

**Key Concepts Added:**

1. **Resolution Algorithm**: Depth-first, left-to-right traversal with last-write-wins merging
   ```
   Given: A extends [B, C], B extends [D], C extends [D, E]
   Visit order: D → B → D(cached) → E → C → A
   ```

2. **Merge Rules by Section**:
   | Section | Behavior |
   |---------|----------|
   | `[tools]` | Deep merge, child overrides parent |
   | `[hooks.*]` | Append (both run, parent first) |
   | `[env.*]` | Deep merge, child overrides |
   | `[services]` | Deep merge, child overrides |

3. **Diamond Dependency Handling**: When multiple paths lead to same config:
   - Loaded only once (cached after first load)
   - Applied only once (first occurrence wins)

4. **Hook Execution Order**: Ancestor-first, left-to-right
   - Enables predictable layering of team → project → local hooks

5. **Error Handling**: Clear error messages for:
   - Circular dependencies: `A → B → C → A`
   - Max depth exceeded (10 levels)
   - Missing parent configs with offline fallback hint
   - Invalid TOML in parent configs

6. **Debug/Inspect Commands**:
   ```bash
   jarvy config show --extends-chain    # Show inheritance chain
   jarvy config validate --check-extends # Validate without running
   jarvy config refresh --recursive      # Force refresh cached parents
   ```

**Updated Technical Implementation**:
- `InheritanceResolver` struct with `in_progress` set for cycle detection
- `resolved_cache` HashMap for diamond dependency optimization
- `ResolutionTrace` for debugging inheritance chains
- Comprehensive merge functions for each config section type

## Tool Dependency Ordering (Completed)

**Date**: 2026-01-17
**Status**: Complete
**Task File**: `tasks/tool-dependency-update.json`

### Overview

Implemented `depends_on` field for tools that have runtime dependencies on other tools. When both tools are in the user's config, Jarvy ensures the dependency is installed first using topological sort ordering.

### Implemented Dependencies

| Tool | Depends On | Reason |
|------|------------|--------|
| node | nvm | Version manager should be installed first |
| python | pyenv | Version manager should be installed first |
| ruby | rbenv | Version manager should be installed first |
| lazydocker | docker | Docker TUI requires Docker daemon |
| kind | docker | Kubernetes-in-Docker requires Docker runtime |
| minikube | docker | Default driver uses Docker |
| kubectl | minikube, kind | Install K8s cluster tools before kubectl |

### Blocked Dependencies (Tools Don't Exist)

| Tool | Would Depend On | Status |
|------|-----------------|--------|
| kotlin | java | Blocked - java tool not in Jarvy |
| scala | java | Blocked - java tool not in Jarvy |
| elixir | erlang | Blocked - erlang tool not in Jarvy |

### Implementation Details

The `depends_on` field in `define_tool!` macro:
```rust
define_tool!(LAZYDOCKER, {
    command: "lazydocker",
    macos: { brew: "lazydocker" },
    linux: { brew: "lazydocker" },
    windows: { choco: "lazydocker" },
    // Docker TUI requires Docker daemon to be installed
    depends_on: &["docker"],
});
```

**Key Code Patterns**:
- `order_tools_by_dependencies()` in `spec.rs` uses Kahn's topological sort
- Dependencies are only considered if the dependency tool is also in the user's config
- If a dependency is not in the config, the tool installs normally (assumes user has it installed elsewhere)

### Key Learnings

1. **Soft Dependencies**: Dependencies are "soft" - they only affect installation order when both tools are in the config
2. **Multiple Dependencies**: Tools can depend on multiple tools (e.g., kubectl depends on both minikube and kind)
3. **No Circular Dependencies**: Topological sort naturally prevents circular dependency issues
4. **Version Manager Pattern**: Language tools (node, python, ruby) depend on their version managers when both are present

### Future Enhancement: PRD-034

Created `prd/034-enhanced-dependency-system.md` to address the limitation that current dependencies are too rigid:

**Problem**: kubectl currently depends on BOTH minikube AND kind, but really just needs ANY Kubernetes cluster (minikube OR kind OR docker desktop OR podman, etc.)

**Solution**: Add `depends_on_one_of` field for flexible dependencies:
```rust
define_tool!(KUBECTL, {
    command: "kubectl",
    // Strict: must have all of these
    depends_on: &[],
    // Flexible: needs at least one of these
    depends_on_one_of: &["minikube", "kind", "docker", "podman"],
});
```

See PRD-034 for full specification.

### Key Technical Decisions

1. **Package Manager Fallbacks**: Tools without native Linux packages use Linuxbrew:
   - checkov, semgrep, dive, nerdctl, trufflehog use `linux: { brew: "..." }`

2. **GUI Apps on Linux**: Some tools (cursor, zed, jetbrains-toolbox) don't have native Linux package manager support. Documentation notes point to official installers.

3. **Windows Coverage**: Limited Windows support for newer tools. Focused on winget where available.

4. **Macro Compliance**: All tools use `define_tool!` macro for consistency and auto-registration via inventory crate.

### Key Learnings

1. **Macro Limitations**: The `define_tool!` macro only supports specific package managers:
   - macOS: `brew`, `cask`
   - Linux: `uniform`, `brew`, or `apt/dnf/pacman/apk` combinations
   - Windows: `winget`, `choco`
   - Not supported: pip, snap, flatpak, cargo, scoop

2. **Linuxbrew Fallback**: For tools without native Linux packages, Linuxbrew is a reliable fallback

3. **GUI App Coverage**: Some popular GUI apps (Cursor, Zed, JetBrains Toolbox) have limited cross-platform package manager availability

4. **Registry Auto-Discovery**: The inventory crate enables zero-config tool registration - just add the module to mod.rs
