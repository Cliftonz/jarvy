# PRD-004: Semantic Version Checking - Implementation Progress

## Started: 2025-01-13
## Completed: 2025-01-13

## Implementation Summary

Successfully implemented proper semantic version checking to replace the buggy substring-based matching.

### Changes Made

1. **Added Dependencies** (Cargo.toml)
   - `semver = "1.0"` - For semantic version parsing and comparison
   - `regex = "1.11"` - For extracting versions from tool output
   - `once_cell = "1.20"` - For lazy static regex compilation

2. **Created New Module** (src/tools/version.rs)
   - `ExtractedVersion` struct to hold parsed version components
   - `extract_version()` - Extracts version from various tool output formats
   - `version_satisfies()` - Main comparison function with semver support
   - `parse_requirement()` - Converts version strings to semver requirements
   - 24 comprehensive unit tests

3. **Updated cmd_satisfies()** (src/tools/common.rs)
   - Now delegates to `version_satisfies()` from the new version module
   - Supports semver operators: `>=`, `<`, `^`, `~`, `=`
   - Supports version ranges: `">= 3.10, < 4.0"`
   - Handles special keywords: `latest`, `*`

### Key Design Decisions

1. **Module Location**: Placed version.rs in `src/tools/` instead of crate root
   - This allows access via `super::version` from common.rs
   - Works for both library and binary compilation

2. **Requirement Parsing Strategy**:
   - Bare versions like "3.10" are NOT passed to semver directly
   - Reason: semver interprets "3.10.5" as "^3.10.5" which is too permissive
   - Instead, we manually convert:
     - "20" → `>=20.0.0, <21.0.0` (major match)
     - "3.10" → `>=3.10.0, <3.11.0` (major.minor match)
     - "3.10.5" → `=3.10.5` (exact match)

3. **Fallback Handling**: Smart substring matching for unparseable versions
   - Uses component-boundary matching to prevent false positives
   - "3.1" won't match "3.10.0" anymore

### Bugs Fixed

1. **False positive substring match**: "2.4" no longer matches "12.40.0"
2. **Embedded number match**: "3.1" no longer matches "3.10.0" or "3.11.0"
3. **Exact version mismatch**: "3.10.5" now correctly rejects "3.10.6"

### Version Formats Supported

| Tool | Output Format | Extracted |
|------|---------------|-----------|
| git | `git version 2.44.0` | 2.44.0 |
| node | `v20.10.0` | 20.10.0 |
| python | `Python 3.12.1` | 3.12.1 |
| docker | `Docker version 24.0.7, build afdd53b` | 24.0.7 |
| rustc | `rustc 1.75.0 (82e1608df 2023-12-21)` | 1.75.0 |
| go | `go version go1.21.5 darwin/arm64` | 1.21.5 |

### Test Results

- 24/24 version-specific tests passing
- All existing tests continue to pass
- Doc tests passing

### Learnings

1. **Rust module system**: Binary and library have separate module trees
   - `mod tools;` in main.rs creates a binary-local module
   - `pub mod tools;` in lib.rs exports from library
   - Use `super::` for relative imports within same tree

2. **Semver crate behavior**: Bare version numbers default to caret (^)
   - `VersionReq::parse("3.10.5")` returns `^3.10.5`, not exact match
   - Must explicitly use `=3.10.5` for exact matching

3. **Regex for version extraction**: Single regex handles most formats
   - Pattern: `v?(\d+)\.(\d+)(?:\.(\d+))?(?:-([a-zA-Z0-9.-]+))?`
   - Handles optional v prefix, optional patch, optional prerelease

# PRD-002: Tool Specification Abstraction - Implementation Progress

## Started: 2025-01-14
## Status: Phase 1 Complete (Proof of Concept)

## Implementation Summary

Successfully implemented the ToolSpec abstraction pattern to eliminate boilerplate across tool implementations. Migrated 5 proof-of-concept tools from ~400 lines to ~100 lines.

### Changes Made

1. **Added Dependencies** (Cargo.toml)
   - `phf = { version = "0.11", features = ["macros"] }` - For compile-time hash maps (future package registry)

2. **Created New Module** (src/tools/spec.rs - 376 lines)
   - `ToolSpec` struct - Declarative tool definition with platform-specific install options
   - `MacOsInstall` struct - Homebrew formula/cask options
   - `LinuxInstall` struct - apt/dnf/yum/zypper/pacman/apk package mappings
   - `WindowsInstall` struct - winget/chocolatey package mappings
   - `CustomInstallFn` type - Function pointer for complex tools needing custom logic
   - `define_tool!` macro - Ergonomic syntax for tool definitions
   - 6 unit tests for struct helpers

3. **Migrated 5 Tools** (git, jq, wget, tree, ripgrep)
   - **Before**: Each tool ~80 lines of boilerplate
   - **After**: Each tool ~25 lines using `define_tool!` macro

### Code Reduction Examples

**Before (jq.rs - 83 lines)**:
```rust
pub fn ensure(_min_hint: &str) -> Result<(), InstallError> {
    if has("jq") { return Ok(()); }
    install()
}
pub fn add_handler(min_hint: &str) -> Result<(), InstallError> { ... }
fn install() -> Result<(), InstallError> {
    #[cfg(target_os = "macos")] { return install_macos(); }
    #[cfg(target_os = "linux")] { return install_linux(); }
    #[cfg(target_os = "windows")] { return install_windows(); }
}
fn install_macos() -> Result<(), InstallError> { ... }
fn install_linux() -> Result<(), InstallError> { ... }
fn install_windows() -> Result<(), InstallError> { ... }
```

**After (jq.rs - 24 lines)**:
```rust
use crate::define_tool;

define_tool!(JQ, {
    command: "jq",
    macos: { brew: "jq" },
    linux: { uniform: "jq" },
    windows: { winget: "jqlang.jq" },
});
```

### Key Design Decisions

1. **Macro vs Struct Only**: Chose hybrid approach
   - Structs provide type safety and IDE autocomplete
   - Macro provides ergonomic syntax for common cases
   - Macro expands to full struct + ensure() + add_handler() functions

2. **Platform Install Abstraction**:
   - `LinuxInstall::uniform(name)` - When all package managers use same name
   - Individual fields for tools with different names per PM
   - `Option<...>` for platforms that don't support the tool

3. **Custom Install Hook**: `custom_install: Option<fn(&str) -> Result<(), InstallError>>`
   - For complex tools like nvm, rustup that need special logic
   - Takes precedence over standard package manager installs

4. **Closure Limitation**: Can't return closure from `into_handler()`
   - Closures capturing `self` can't be coerced to `fn` pointers
   - Solution: Macro generates `ensure()` and `add_handler()` directly

### Macro Syntax Variants Supported

```rust
// Homebrew formula
macos: { brew: "git" }

// Homebrew cask (for GUI apps)
macos: { cask: "visual-studio-code" }

// Uniform Linux package name
linux: { uniform: "git" }

// Full Linux specification
linux: { apt: "git", dnf: "git", pacman: "git", apk: "git" }

// Windows winget
windows: { winget: "Git.Git" }

// No Windows support (built-in like tree.exe)
// Simply omit the windows: block
```

### Test Results

- 75/75 library tests passing
- All 5 migrated tools work correctly
- 6 new spec module tests passing

### Learnings

1. **Rust Macros**: Declarative macros (`macro_rules!`) work well for this pattern
   - Use `$($key:ident: $val:expr),*` for optional repeated fields
   - Helper rules (@macos, @linux, etc.) make complex macros manageable
   - `stringify!($name)` converts identifier to string literal

2. **Static Lifetime**: `ToolSpec` uses `&'static str` throughout
   - Allows static definition: `pub static GIT: ToolSpec = ...`
   - Zero runtime allocation
   - Works with `const fn` constructors

3. **Conditional Compilation**: `#[cfg(target_os)]` in impl blocks
   - Can have different methods for different platforms
   - Use `#[cfg(target_os = "linux")]` on imports too

4. **Function Pointers vs Closures**: Key limitation
   - `fn(&str) -> Result<(), E>` is a function pointer type
   - Closures capturing variables have different, non-coercible types
   - Solution: Generate concrete functions via macro

## T7 Completed: 2025-01-14

### Auto-Registration with Inventory Crate

Successfully implemented automatic tool registration using the `inventory` crate,
eliminating the need for manual registration in `register_all()`.

### Implementation Details

1. **Added Dependency** (Cargo.toml)
   - `inventory = "0.3.21"` - For compile-time collection of tool entries

2. **Created Registration Infrastructure** (src/tools/spec.rs)
   - `ToolEntry` struct - Wrapper containing ToolSpec reference and handler function
   - `inventory::collect!(ToolEntry)` - Enables compile-time collection
   - `iter_tools()` function - Iterator over all registered ToolEntry instances

3. **Updated define_tool! Macro** (src/tools/spec.rs)
   - Added `::inventory::submit!` block inside macro expansion
   - Each tool automatically submits its ToolEntry when the macro expands
   - No additional code needed in individual tool files

4. **Updated register_all()** (src/tools/mod.rs)
   - Replaced 36+ manual registration lines with single loop:
     ```rust
     for entry in spec::iter_tools() {
         let _ = register_tool(entry.spec.name, entry.handler);
     }
     ```
   - Only 3 custom-installer tools (nvm, rust, brew) remain manually registered

### Why Inventory Over Linkme

**Chosen: `inventory` crate**
- Simpler API: `inventory::submit!` + `inventory::iter::<T>`
- Works naturally with static structs
- Well-maintained with 13M+ downloads
- Clean integration with macro_rules!

**Rejected: `linkme` crate**
- More complex distributed slice API
- Requires explicit slice definition
- More platform-specific linker considerations

### Acceptance Criteria Verification

- ✅ Tools register themselves at compile time (via `inventory::submit!` in macro)
- ✅ No manual edits to register_all() needed for new tools
- ✅ register_all() discovers all ToolSpec definitions automatically
- ✅ Works on all target platforms (macOS, Linux, Windows) - inventory is cross-platform
- ✅ All existing tests continue to pass (180+ tests passing)

### Key Learnings

1. **Inventory Pattern**: The crate uses linker magic to collect statics at compile time
   - Works across crate boundaries
   - Zero runtime cost after initial collection

2. **Macro Integration**: `::inventory::submit!` must use fully-qualified path
   - Prevents import conflicts in expansion context
   - `$crate::` prefix needed for internal types

3. **ToolEntry Wrapper**: Needed to bundle spec + handler together
   - Can't collect raw ToolSpec because handler function is generated per-tool
   - Handler function pointers stored alongside spec references

### Hook Configuration Format

```toml
[provisioner]
git = "latest"
node = "20"

[hooks]
pre_setup = "echo 'Starting setup...'"
post_setup = "echo 'Setup complete!'"

[hooks.config]
shell = "bash"
timeout = 300
continue_on_error = false

[hooks.git]
post_install = "git config --global init.defaultBranch main"

[hooks.node]
post_install = "npm install -g pnpm"
```

### Environment Variables Injected

| Variable | Description |
|----------|-------------|
| `JARVY_TOOL` | Tool name (for per-tool hooks) |
| `JARVY_VERSION` | Version being installed |
| `JARVY_OS` | Operating system (macos, linux, windows) |
| `JARVY_ARCH` | CPU architecture (x86_64, aarch64) |
| `JARVY_HOME` | Jarvy home directory (~/.jarvy) |

### Key Design Decisions

1. **Shell Detection**: Auto-detect from `$SHELL` env var on Unix, PowerShell on Windows
   - Supports bash, zsh, sh, fish, PowerShell, cmd

2. **Timeout Implementation**: Used `wait-timeout` crate for cross-platform timeout
   - Default 300 seconds (5 minutes)
   - Process killed if timeout exceeded

3. **Streaming Output**: Hook stdout/stderr captured and printed in real-time
   - Uses separate threads for stdout and stderr
   - Output prefixed with indentation for clarity

4. **TOML Parsing**: Used `#[serde(flatten)]` for tool_hooks HashMap
   - Allows `[hooks.git]`, `[hooks.node]` sections
   - Filters out `config` key from tool names

5. **Error Handling**: Respects `continue_on_error` setting
   - Default: false (stop on first failure)
   - When true: warn and continue

### Shell Command Building

Cross-platform shell execution:
- **Unix shells**: `{shell} -c "{script}"`
- **Fish**: `fish -c "{script}"`
- **PowerShell**: `powershell -NoProfile -Command "{script}"`
- **CMD**: `cmd /C "{script}"`

### Learnings

1. **Serde Flatten**: `#[serde(flatten)]` captures extra keys into HashMap
   - Perfect for dynamic `[hooks.{tool}]` sections
   - Must filter out known keys (like "config")

2. **Wait-Timeout Crate**: Cross-platform process timeout

3. **Thread-based Output Capture**: Can't use async for stdout/stderr
   - BufReader::lines() is blocking
   - Spawn threads to capture output while waiting for timeout

### Implementation Details

All 35 tools were implemented using the `define_tool!` macro pattern:

```rust
define_tool!(TOOLNAME, {
    command: "tool-command",
    macos: { brew: "package" },
    linux: { uniform: "package" },  // or { apt: "x", dnf: "y", ... }
    windows: { winget: "Publisher.Package" },
});
```
