# PRD-016: Developer Experience Commands

## Overview

Add a suite of commands that make Jarvy more useful for daily development workflows, including environment diagnostics, dry-run previews, tool discovery, and shell completions.

## Problem Statement

Jarvy currently focuses on initial environment setup, but developers need ongoing tools for:
- Diagnosing why tools aren't working (PATH issues, version conflicts)
- Previewing changes before running setup (avoiding surprises)
- Discovering what tools are available
- Generating configurations from existing setups
- Keeping tools up to date
- Shell integration for faster command entry

Users currently must manually investigate issues, read documentation to find available tools, and lack visibility into what `jarvy setup` will actually do.

## Evidence

- Common support questions: "Why isn't [tool] found after setup?"
- No way to preview what setup will change
- New users don't know what tools are available
- Teams want to generate jarvy.toml from existing environments
- No shell completion support despite being a CLI tool

## Requirements

### Functional Requirements

1. **`jarvy doctor`**: Diagnose environment health
2. **`jarvy diff`**: Preview changes before setup (dry-run)
3. **`jarvy export`**: Generate jarvy.toml from installed tools
4. **`jarvy upgrade`**: Upgrade all tools to latest versions
5. **`jarvy search`**: Search available tools
6. **Shell completions**: bash, zsh, fish, PowerShell

### Non-Functional Requirements

1. All commands support `--format` flag (human, json, quiet)
2. Commands are fast (< 1s for non-network operations)
3. Clear, actionable output for each command
4. Works in CI mode with appropriate defaults

## Non-Goals

- Interactive TUI (terminal user interface)
- GUI or web interface
- Real-time monitoring dashboards
- Automatic background updates

## Command Specifications

### 1. `jarvy doctor`

Diagnose environment issues, check tool health, and verify PATH configuration.

```bash
# Basic health check
jarvy doctor

# Output:
# Jarvy Doctor
# ============
#
# System Information:
#   OS: macOS 14.2 (arm64)
#   Shell: /bin/zsh
#   Package Manager: Homebrew 4.2.0
#
# PATH Analysis:
#   [OK] /opt/homebrew/bin is in PATH
#   [OK] ~/.cargo/bin is in PATH
#   [WARN] ~/.nvm/current/bin not in PATH (nvm may not work)
#
# Tool Health:
#   [OK] git 2.43.0 (installed: 2.43.0) - matches requirement
#   [OK] node 20.10.0 (installed: 20.11.0) - satisfies >=20
#   [WARN] rust 1.75.0 (installed: 1.70.0) - outdated
#   [ERROR] docker - not found in PATH
#
# Hooks Status:
#   [OK] nvm shell integration active
#   [WARN] starship not initialized in .zshrc
#
# Recommendations:
#   1. Install docker: jarvy setup --only docker
#   2. Update rust: jarvy upgrade rust
#   3. Add starship init: echo 'eval "$(starship init zsh)"' >> ~/.zshrc

# JSON output for scripting
jarvy doctor --format json

# Check specific tools only
jarvy doctor git node rust

# Quiet mode for CI (exit code only)
jarvy doctor --format quiet
# Exit 0: all healthy
# Exit 1: warnings present
# Exit 2: errors present
```

**Checks performed:**
- System information (OS, shell, package managers)
- PATH correctness (expected directories present)
- Tool installation status (installed vs required)
- Version satisfaction (installed version meets requirement)
- Hook/integration status (shell rc files configured)
- Common misconfiguration detection

### 2. `jarvy diff`

Show what would change before running setup (dry-run preview).

```bash
# Preview all changes
jarvy diff

# Output:
# Jarvy Diff - Preview of Changes
# ================================
#
# Tools to Install:
#   + docker (latest) via homebrew cask
#   + kubectl (1.29) via homebrew
#
# Tools to Update:
#   ~ rust 1.70.0 -> 1.75.0 (requires: 1.75)
#   ~ node 18.0.0 -> 20.10.0 (requires: 20)
#
# Tools Already Satisfied:
#   = git 2.43.0 (requires: >=2.40)
#   = jq 1.7.1 (requires: latest)
#
# Hooks to Run:
#   -> nvm: Configure shell integration
#   -> starship: Add shell initialization
#
# Services to Start:
#   -> postgres (docker-compose up)
#
# No changes will be made. Run 'jarvy setup' to apply.

# JSON output
jarvy diff --format json

# Compare against different config
jarvy diff --config path/to/other/jarvy.toml

# Show only changes (hide satisfied)
jarvy diff --changes-only
```

**Information shown:**
- Tools that will be installed (with method)
- Tools that will be updated (old -> new version)
- Tools already satisfied (no action needed)
- Hooks that will execute
- Services that will start/stop
- Environment variables that will be set

### 3. `jarvy export`

Generate jarvy.toml from currently installed tools on the system.

```bash
# Export detected tools
jarvy export

# Output:
# # Generated by jarvy export on 2024-01-15
# # Detected tools from current environment
#
# [tools]
# git = "2.43"
# node = "20.11"
# rust = "1.75"
# python = "3.12"
# go = "1.21"
# docker = "latest"
# jq = "1.7"

# Write to file
jarvy export > jarvy.toml
jarvy export --output jarvy.toml

# Export specific tools only
jarvy export git node rust

# Include all detected tools (even uncommon ones)
jarvy export --all

# Include version managers and their versions
jarvy export --include-version-managers

# JSON output
jarvy export --format json

# Include comments with install locations
jarvy export --verbose
# Output:
# [tools]
# git = "2.43"     # /opt/homebrew/bin/git
# node = "20.11"  # ~/.nvm/versions/node/v20.11.0/bin/node
```

**Export behavior:**
- Detect tools that Jarvy supports
- Extract current versions
- Generate valid jarvy.toml syntax
- Optionally include comments with paths
- Handle version managers (nvm, pyenv, rbenv)

### 4. `jarvy upgrade`

Upgrade all tools to their latest versions.

```bash
# Upgrade all tools
jarvy upgrade

# Output:
# Upgrading tools...
#
# [1/3] git: 2.43.0 -> 2.44.0
#       brew upgrade git... done
#
# [2/3] node: 20.10.0 -> 20.11.0
#       nvm install 20.11.0... done
#       nvm alias default 20.11.0... done
#
# [3/3] rust: 1.75.0 -> 1.76.0
#       rustup update stable... done
#
# Upgrade complete. 3 tools updated.

# Upgrade specific tools
jarvy upgrade git node

# Dry run (show what would upgrade)
jarvy upgrade --dry-run

# Force upgrade even if version constraint satisfied
jarvy upgrade --force

# Upgrade to specific version
jarvy upgrade node@22

# JSON output
jarvy upgrade --format json

# Quiet mode
jarvy upgrade --format quiet
```

**Upgrade behavior:**
- Check for newer versions available
- Use appropriate upgrade method per tool
- Handle version managers correctly
- Update jarvy.toml if using "latest" constraint
- Run post-install hooks after upgrade
- Support selective upgrades

### 5. `jarvy search <query>`

Search available tools that Jarvy can install.

```bash
# Search by name
jarvy search docker

# Output:
# Available Tools Matching "docker"
# =================================
#
# docker
#   Container runtime for building and running applications
#   Platforms: macOS (cask), Linux (apt/dnf/pacman), Windows (winget)
#   Categories: containers, devops
#
# docker-compose
#   Define and run multi-container Docker applications
#   Platforms: macOS (brew), Linux (apt/dnf), Windows (winget)
#   Categories: containers, devops
#
# lazydocker
#   Terminal UI for Docker and Docker Compose
#   Platforms: macOS (brew), Linux (brew), Windows (scoop)
#   Categories: containers, tui

# Search by category
jarvy search --category databases

# List all available tools
jarvy search --all
jarvy tools --list

# JSON output
jarvy search docker --format json

# Show install command
jarvy search docker --show-install
# To install: jarvy setup --only docker
# Or add to jarvy.toml:
#   [tools]
#   docker = "latest"
```

**Search features:**
- Fuzzy name matching
- Category filtering
- Platform availability info
- Description and metadata
- Installation hints

### 6. Shell Completions

Generate shell completions for bash, zsh, fish, and PowerShell.

```bash
# Generate completions
jarvy completions bash > /usr/local/etc/bash_completion.d/jarvy
jarvy completions zsh > ~/.zsh/completions/_jarvy
jarvy completions fish > ~/.config/fish/completions/jarvy.fish
jarvy completions powershell > jarvy.ps1

# Installation instructions
jarvy completions --install-instructions

# Output:
# Shell Completion Installation
# =============================
#
# Bash:
#   jarvy completions bash > /usr/local/etc/bash_completion.d/jarvy
#   source ~/.bashrc
#
# Zsh:
#   jarvy completions zsh > ~/.zsh/completions/_jarvy
#   # Add to .zshrc: fpath=(~/.zsh/completions $fpath)
#   autoload -Uz compinit && compinit
#
# Fish:
#   jarvy completions fish > ~/.config/fish/completions/jarvy.fish
#
# PowerShell:
#   jarvy completions powershell >> $PROFILE
```

**Completions include:**
- All commands and subcommands
- Command-specific flags and options
- Tool names for relevant commands
- File path completion where appropriate

### Doctor Implementation

```rust
// src/commands/doctor.rs
pub struct DoctorResult {
    pub system: SystemInfo,
    pub path_checks: Vec<PathCheck>,
    pub tools: Vec<ToolHealth>,
    pub hooks: Vec<HookStatus>,
    pub recommendations: Vec<Recommendation>,
}

pub fn run_doctor(tools: Option<Vec<String>>) -> Result<DoctorResult, Error> {
    let system = collect_system_info()?;
    let path_checks = check_path_entries()?;
    let tools = check_tool_health(tools)?;
    let hooks = check_hook_status()?;
    let recommendations = generate_recommendations(&path_checks, &tools, &hooks);

    Ok(DoctorResult { system, path_checks, tools, hooks, recommendations })
}
```

### Diff Implementation

```rust
// src/commands/diff.rs
pub struct DiffResult {
    pub to_install: Vec<ToolChange>,
    pub to_update: Vec<ToolChange>,
    pub satisfied: Vec<ToolStatus>,
    pub hooks_to_run: Vec<HookInfo>,
    pub services_to_start: Vec<ServiceInfo>,
}

pub fn run_diff(config: &Config) -> Result<DiffResult, Error> {
    let current_state = detect_current_state()?;
    let desired_state = parse_desired_state(config)?;

    Ok(compute_diff(current_state, desired_state))
}
```

### Shell Completions with clap

```rust
// src/commands/completions.rs
use clap_complete::{generate, Shell};

pub fn generate_completions(shell: Shell) -> String {
    let mut cmd = build_cli();
    let mut buf = Vec::new();
    generate(shell, &mut cmd, "jarvy", &mut buf);
    String::from_utf8(buf).unwrap()
}
```

## Risks

1. **Version detection complexity**: Different tools report versions differently
   - Mitigation: Centralize version parsing, handle edge cases

2. **Cross-platform PATH handling**: PATH works differently across OS
   - Mitigation: Platform-specific PATH resolution logic

3. **Shell integration detection**: Hard to detect if hooks are active
   - Mitigation: Check for common patterns in rc files

4. **Upgrade safety**: Upgrades could break user environments
   - Mitigation: Always show dry-run first, prompt for confirmation
